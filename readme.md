1. Введение и описание проекта
TopLook — это веб-приложение, представляющее собой социальную платформу для обмена модными образами (аутфитами). Главная цель проекта — создание надежной, масштабируемой и безопасной системы с использованием современных паттернов проектирования и фреймворка Java Spring Boot. 
В приложении реализован полный цикл взаимодействия пользователей:
•	Регистрация и защищенная аутентификация. 
•	Создание публикаций (постов) с описанием и фотографиями. 
•	Социальные взаимодействия: возможность ставить лайки, оставлять комментарии и подписываться на других пользователей. 
•	Формирование персональной ленты новостей (Feed) на основе подписок. 
2. Технологический стек
Для реализации проекта был выбран следующий стек технологий, отвечающий современным стандартам веб-разработки:
•	Backend: Java 17, Spring Boot 3.3.1 (модули Web, Data JPA, Security, Validation). 
•	База данных: PostgreSQL с использованием ORM (Hibernate) для автоматической генерации таблиц и запросов. 
•	Безопасность: Spring Security в связке с JSON Web Tokens (JWT) для реализации авторизации без сохранения состояния сессии (stateless). 
•	Frontend: HTML5, CSS3, JavaScript (Vanilla JS), интеграция с сервером через Fetch API. 
•	Сборка и зависимости: Gradle. 
•	Документирование API: SpringDoc OpenAPI (Swagger UI). 
3. Архитектура системы
Проект построен по строгой многоуровневой архитектуре (Layered Architecture), что обеспечивает разделение ответственности и легкость поддержки кода.
Система состоит из следующих слоев:
1.	Слой представления (Frontend): Клиентская часть отправляет асинхронные HTTP-запросы к REST API сервера. 
2.	Слой контроллеров (Controllers): Принимает входящие запросы, валидирует данные с помощью аннотаций (например, @Valid) и маршрутизирует их. Включает в себя AuthController, PostController, SocialController и UserController. 
3.	Слой бизнес-логики (Services): Выполняет основные операции приложения. Например, PostService отвечает за формирование ленты новостей, а SocialService — за обработку подписок и лайков. 
4.	Слой доступа к данным (Repositories): Интерфейсы, наследующие JpaRepository, отвечают за взаимодействие с базой данных PostgreSQL без написания ручных SQL-запросов. 
 
4. Проектирование базы данных
База данных спроектирована в соответствии с правилами нормализации для исключения избыточности данных. Схема включает 5 основных сущностей:
•	users: Хранит идентификатор, логин, email, зашифрованный пароль, биографию и ссылку на аватар. 
•	posts: Хранит информацию о публикациях (описание, URL изображения, категория) и имеет внешний ключ (Foreign Key) author_id, связывающий пост с создателем. 
•	comments: Хранит текст комментария и связывает пользователя (user_id) с конкретным постом (post_id). 
•	likes: Промежуточная таблица для реализации связи «Многие-ко-многим» между пользователями и постами (кто что лайкнул). 
•	follows: Промежуточная таблица для реализации системы подписок, связывающая подписчика (follower_id) и того, на кого подписались (following_id). 
 5. Безопасность и DTO (Data Transfer Object)
Аутентификация (JWT): В приложении отключены стандартные серверные сессии (SessionCreationPolicy.STATELESS). Вместо этого используется механизм JWT. При успешном входе пользователя (AuthController.login) сервер генерирует криптографически подписанный токен. Клиент сохраняет его в браузере и прикрепляет к заголовку Authorization: Bearer при каждом последующем запросе. Кастомный фильтр JwtAuthenticationFilter перехватывает запросы и проверяет подлинность токена. Пароли пользователей необратимо хэшируются с использованием BCryptPasswordEncoder. 
Паттерн DTO: Для передачи данных между сервером и клиентом используются специальные объекты DTO (например, UserDto, PostDto). Это исключает возможность случайной передачи конфиденциальных данных (таких как хэш пароля) на клиентскую сторону и позволяет объединять данные из разных таблиц в один удобный JSON-ответ. 
6. Глобальная обработка исключений
Для повышения отказоустойчивости API внедрен класс GlobalExceptionHandler, использующий аннотацию @RestControllerAdvice. Он перехватывает системные исключения (например, IllegalArgumentException, SecurityException) и возвращает клиенту стандартизированный JSON-ответ с HTTP-статусом (400 Bad Request, 403 Forbidden, 409 Conflict) и понятным сообщением об ошибке. Это предотвращает падение сервера и предоставляет фронтенду точную информацию для отображения уведомлений пользователю.
